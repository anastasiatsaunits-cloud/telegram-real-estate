# Архитектура проекта: Telegram-бот + Mini App каталога недвижимости

## 1. Цель проекта

Сделать отдельный цифровой продукт внутри Telegram, состоящий из:
- Telegram-бота как входной точки,
- Telegram Mini App как основного интерфейса,
- backend-сервиса с бизнес-логикой,
- CRM-интеграции,
- аналитики,
- генерации подборок и PDF,
- каталога объектов недвижимости.

Проект должен повторять логику и уровень UX референса со скриншотов:
- вход через бота,
- открытие каталога внутри Telegram,
- прохождение мини-квиза,
- показ персональной подборки,
- просмотр карточек объектов,
- инвестиционные расчёты,
- захват лида через телефон,
- передача лида в CRM.

---

## 2. Продуктовая схема

### Основной пользовательский сценарий

1. Пользователь заходит в Telegram-бот
2. Получает welcome-сообщение с оффером
3. Нажимает кнопку открытия каталога
4. Открывается Telegram Mini App
5. Пользователь проходит квиз:
   - выбор региона,
   - выбор бюджета,
   - выбор срока покупки,
   - при необходимости дополнительные параметры
6. Система формирует подборку объектов
7. Пользователь смотрит список объектов
8. Открывает карточку объекта
9. Изучает параметры, доходность, ROI, способы покупки
10. Оставляет телефон / заявку / запрос PDF
11. Лид передаётся в CRM
12. Менеджер получает заявку и продолжает продажу

Проект состоит из 5 основных частей:

Роль:

Роль:
- основной интерфейс продукта,
- квиз,

- работа с лидами,
- генерация расчётов,
- CRM-интеграция,
- аналитика,
- контентные блоки.

### 3.4 База данных
Роль:
- пользователи,
- объекты,
- ответы квиза,
- лиды,
- события,
- инвестиционные метрики,
- контент.

- Telegram Bot API

- CRM (приоритетно Bitrix24)

## 6. Telegram Mini App

### Основная роль
Mini App должен быть главным продуктовым интерфейсом. Именно здесь происходит выбор параметров, просмотр объектов и конверсия в лид.

### Основные разделы

#### 6.1 Главная / витрина
Содержит:
- баннеры,
- категории регионов,
- кнопку `Подобрать объект`,
- рекламные карточки,
- вход в каталог,
- кнопки перехода в канал / отзывы / о компании / вакансии.

#### 6.2 Quiz / подбор объекта
Шаги квиза, видимые по скринам:
- регион,
- бюджет,
- срок покупки.

Дополнительно стоит предусмотреть:
- цель покупки,
- тип объекта,
- способ оплаты,
- инвестиционная цель,
- желаемый доход,
- площадь / число комнат.

#### 6.3 Результат подбора
- экран `Подборка готова`
- CTA `Получить подборку`
- переход к вводу телефона

#### 6.4 Захват лида
- экран ввода телефона
- валидация номера
- согласие на обработку данных
- передача данных в CRM

#### 6.5 Каталог
- список объектов,
- карточки,
- фильтры,
- сортировка,
- быстрые переходы по регионам и типам.

#### 6.6 Карточка объекта
Содержит:
- фотогалерею,
- название,
- локацию,
- цену от,
- описание проекта,
- площадь,
- цели покупки,
- варианты оплаты,
- инвестиционный блок,
- CTA `Оставить заявку`,
- CTA `Получить полный расчёт PDF`.

#### 6.7 Инвестиционный блок
По скринам видно, что в карточке есть:
- аренда % годовых,
- краткосрочная доходность,
- рост цены в год,
- ROI 5 лет,
- ROI 10 лет,
- альтернативный сценарий,
- итоговые прогнозные значения,
- CTA на PDF.

### Технический стек
- React
- TypeScript
- Next.js или Vite
- Tailwind CSS
- Telegram WebApp SDK
- Zustand / Redux Toolkit
- React Hook Form + zod

## 7. Backend API

### Основные модули

#### 7.1 Auth / Telegram Session
- проверка initData
- идентификация пользователя
- создание сессии
- определение источника трафика

#### 7.2 Catalog Service
- регионы
- города
- проекты
- фильтрация
- сортировка
- карточка объекта
- подборка объектов под ответы квиза

#### 7.3 Quiz Service
- сценарии шагов
- хранение ответов
- ветвление логики
- формирование профиля спроса пользователя

#### 7.4 Recommendation Service
- подбор релевантных объектов
- расчёт порядка выдачи

#### 7.5 Lead Service
- приём контакта
- создание лида
- проверка дублей
- статусы лида
- передача в CRM

#### 7.6 Investment Calculator
- модель доходности
- ROI 5 / 10 лет
- рост цены
- аренда
- альтернативный сценарий
- расчёт прогнозной стоимости

#### 7.7 PDF Service
- формирование полного расчёта
- генерация PDF
- отправка PDF пользователю
- сохранение истории запросов

#### 7.8 CRM Integration Service
- создание лида в Bitrix24
- обновление лида
- передача параметров квиза
- передача просмотренных объектов
- передача источника лида

#### 7.9 Analytics Service
- сбор событий воронки
- расчёт конверсии
- отчёты по шагам
- сегментация по источникам и регионам

### Технический стек
- NestJS или Fastify
- REST API
- Redis + очередь задач для PDF и отложенных действий

## 17. Ключевые риски и что предусмотреть заранее

### Продуктовые риски
- слишком ранний запрос телефона
- слабая ценность до лида
- сложный или длинный квиз
- некачественная карточка объекта

### Технические риски
- невалидная интеграция Telegram WebApp
- плохая производительность mini app
- дубли лидов в CRM
- разъезд данных между каталогом и CRM
- ручное обновление контента через разработчика

### Что нужно предусмотреть сразу
- идемпотентность при создании лидов
- централизованный расчёт ROI
- нормальную схему хранения объектов и медиа
- событийную аналитику по каждому шагу
- отдельную среду staging

## 19. Итог

Рекомендация: делать это как **отдельный единый продукт в monorepo**, а не как набор разрозненных частей.

Оптимальная архитектура:
- Telegram-бот как входной и коммуникационный слой,
- Mini App как основной продуктовый интерфейс,
- backend как единый центр логики,
- PostgreSQL как основное хранилище,
- Redis для фоновых задач,
- Bitrix24 как CRM-приёмник,
- аналитика событий для управления конверсией.

Такой подход позволит:
- быстро запустить MVP,
- дальше наращивать функциональность,
- контролировать воронку,
- масштабировать каталог,
- управлять контентом без постоянной разработки.
